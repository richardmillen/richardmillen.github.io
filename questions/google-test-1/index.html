The MATCHER_P macro in Google Test defines a custom matcher that accepts a single argument.

2.

  1. Write a test parameter class or struct that will contain the test case data along with expected results.
  2. Write a test fixture class that inherits from TestWithParam<T> where T is your test parameter class.
  3. Write a new test, using the TEST_P macro.
  4. Inside the test, call GetParam() to return an instance of your test parameter.
  5. Define a global array of test parameter objects (possibly immediately below the test).
  6. INSTANTIATE_TEST_CASE_P(<test-name-appendage>, <fixture-name>, ValuesIn(<array-of-values>)) - This invokes the test.

Quiz 4

  1. How would you define a mock?
  2. How do you configure Google Mock to verify that a method is called on a test double? Give an example.
  3. How do you force Google Mock to run its verification before a mock goes out of scope?
  4. How do you get Google Mock to match on any call to a method, regardless of the argument?
  5. How do you tell Google Mock that a method will return the string “ABC” once (and only once)?

1.

  • Derive Mock<interface-name> from <interface-name>.
  • Perform the following steps for each virtual function you want to mock:
    • Count how many arguments it has.
    • In the ‘public:’ section write MOCK_METHODn(); or MOCK_CONST_METHODn(); where ‘n’ is the number of arguments.
    • Copy & paste the function signature between the parentheses of the macro. Cut the function name; that becomes the 1st parameter, what’s left becomes the 2nd parameter (this is the type of the function).

2.

Use the EXPECT_CALL macro.
e.g.

EXPECT_CALL(mock, method_name(arg1, arg2, etc));
// perform some action...

3.

Mock::VerifyAndClearExpectations(&mock);

4.

Use the EXPECT_CALL macro, passing the wildcard matcher testing::_ as the method argument.
e.g.

EXPECT_CALL(mock, method_name(_));

5.

Use the EXPECT_CALL macro. Invoke the WillOnce method on the returned TypedExpectation, passing in a ReturnAction with the expected string.
e.g.

EXPECT_CALL(...).WillOnce(Return("ABC"));

Quiz 6

  1. You can specify one action for a Google Mock expectation using either WillOnce() or WillRepeadedly(). What if a function needs to do more?
  2. What does the Google Mock action SetArgPointee do?
  3. How and why might you run a subset of tests?
  4. Explain the lifecycle of a test fixture object.
  5. When should you use a the constructor/destructor of a test fixture to perform per-test set-up and tear-down logic? What other options are there and when should they be used?

1.

Use DoAll() which creates a composite action from other actions.

2.

Tells Google Mock to set up the pointee (the thing being poited at) for the n’th argument to a specified value.
e.g.

SetArgPointee<n>(value)

3.

test-program --gtest_filter=<filter>

Saves execution time.

4.

For each TEST_F, Google Test will create a fresh test fixture object, immediately call SetUp(), run the test body, call TearDown(), and then immediately delete the test fixture object.

5.

  • Use of constructor/destructor is preferred because:-
    • You have the option to make variables const.
    • They will be called even if you need to subclass the fixture class.
  • Use SetUp()/TearDown() methods in the following cases:-
    • If tear-down operation could throw an exception.
    • When using assertions during tear-down and the --gtest_throw_on_failure flag is specified.
    • If you need to call a method that will be overriden in a derived class. Calls to virtual functions on ‘this’ are statically bound in a constructor/destructor.
