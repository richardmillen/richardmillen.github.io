Define the set of methods that are associated with values or pointers of a given type.
If you implement an interface using a pointer receiver, then only pointers of that type implement the interface.
If you implement an interface using a value receiver, then both values and pointers of that type implement the interface.

5.

Interface values are two-word data structures.
The first word contains a pointer to an internal table called an iTable, which contains type information about the stored value (concrete type).
The iTable contains the type of value that has been stored and a list of methods associated with the value.
The second word is a pointer to the stored value (concrete type).

Take an existing type, declare it within a new struct type.
The embedded type is called an ‘inner’ type of the new ‘outer’ type.
Identifiers may be promoted up to the outer type.
i.e.

The fields of the inner type may NOT be initialised as part of the initialisation of the outer type.

2.

Communicating Sequential Processes.
Message-passing model that works by communicating data between goroutines instead of locking data to synchronize access.

3.

OS schedules threads against physical processors
Go runtime schedules goroutines against logical processors.
Each logical processor is bound to a single thread.
As of v1.5, default is to allocate a logical processor for every physical processor.

4.

Hundreds of thousands of goroutines can be scheduled to run concurrently with amazing efficiency and performance.

5.

The thread and goroutine are detached from the logical processor.
The thread continues to block.
The scheduler creates a new thread and attaches it to the logical processor.
The scheduler chooses another goroutine from local run queue for execution.
Once the syscall returns the goroutine is placed back into a local run queue.
The now unblocked thread is put aside for future use.

func New(maxGoroutines int) *Pool {
    p := Pool{
        work: make(chan Worker),
    }
    p.wg.Add(maxGoroutines)
    for i := 0; i < maxGoroutines; i++ {
        go func() {
            for w := range p.work {
                w.Task()
            }
            wg.Done()
        }()
    }
    return &p
}
func (p *Pool) Run(w Worker) {
    p.work <- w
}
func (p *Pool) Shutdown() {
    close(p.tasks)
    p.wg.Wait()

  • Function named BenchmarkFoo instead of TestFoo.
  • Takes single parameter b *testing.B instead of t *testing.T.
  • In order for benchmark to calculate performance it must run the code over and over again for a period of time. e.g.
func BenchmarkFoo(b *testing.B) {
    // do set ups - this might take a while
    
    b.ResetTimer()
    
    for i := 0; i < b.N; i++ {
        // perform benchmark test here...
    }
}

3.

By default the benchmarking framework will call the benchmark function over and over again for at least one second.

4.

Each time the framework calls the benchmark function, it will increase the value of b.N.
On the first call, the value of b.N will be 1.
All code must be placed inside a loop that repeats b.N times.

5.

ns/op - nanoseconds per operation.
e.g.
