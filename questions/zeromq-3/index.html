ZeroMQ Quiz, Part 3

! W O R K ~ I N ~ P R O G R E S S !

This page contains questions & answers based on the ZeroMQ Guide written by Pieter Hintjens.

Quiz 1

  1. What aspects of control vs data flow when designing a protocol?
  2. What aspect of messaging is important when talking about performance?
  3. What aspect of messaging is affected by ‘the cost of serialisation’?
  4. What are some examples of simple self-describing structured encodings?
  5. What is the ‘Cheap’ part of a protocol?

1.

”> Control dialogue tends to last a short time, involing few messages. > The data flow could last for hours or days, involing billions of messages. > Control is where ““normal”” errors happen e.g. not authenticated, not found, censored etc. > Errors during data flow are exceptional (disk full, server crashed). > Control is where things will change over time, with new options, parameters etc. > The data flow barely changes because the semantics of a resource are fairly constant over time. > Control dialogue is akin to REQ-REP. Data flow is essentially one-way asynchronous flow.”

2.

Data flows.

3.

Data flows.

4.

”> XML > JSON > HTTP-style headers”

5.

”> Control messaging encoding. > Uses any encoding that has a simple, standard parser for it in the target language. > Uses straight request-reply model, where each reply has a success/failure reply. > It doesn’t try to be fast. > Shouldn’t crash, or leak memory. > Should be highly tolerant.”

Quiz 2

  1. What is the ‘Nasty’ part of a protocol?
  2. What is the only thing that matters when design the ‘Nasty’ part of a protocol?
  3. How does the ‘Cheap or Nasty’ pattern simplify error handling?
  4. What is the simplest and most widely used serialisation format for 0MQ apps?
  5. What should you do when using 0MQ’s own multipart framing as a serialisation format?

1.

”> Carries minimal info that barely changes. > Asynchronous, terse, silent and inflexible. > Ultra-fast to parse. > Possibly impossible to extend / experiement with. > Hand-optimised binary layout. > Pure asynchronous model, without acknowledgements, or ‘credit-based flow control’. > Rejects anything it doesn’t like. > Zero memory allocs. > Never crashes.”

2.

Performance.

3.

”> Synchronous control commands: ~ Errors are normal. ~ Each request has a response. > Asynchronous data commands: ~ Errors are exceptional. ~ Bad commands are either discarded silently, or cause disconnect.”

4.

ZeroMQ’s own multi-part framing.

5.

Split messages into a ‘Cheap’ part and a ‘Nasty’ part, using framing to separate them.

Quiz 3

  1. Give an example of how you might separate the ‘Cheap’ from the ‘Nasty’ when using 0MQ’s own multipart framing.
  3. How does a ‘Cheap’ serialisation language enable experimentation?

1.

“Frame 0: Frame 1: command header (for intermediaries). Frame 2: command body (for the application).”

2.

These [XML, JSON, HTTP-style headers] are all pretty equivalent as long as you don’t go overboard with validating parsers, schemas, and other “trust us, this is all for your own good” nonsense.

3.

”> Ignore any elements/attributes/headers that you don’t recognize. > Simple to write generic parsers that thunk a command to/from a hash table.”

4.

If your universe is a scripting language, then go for JSON. If you are aiming to build protocols for wider system use, keep things simple for C developers and stick to HTTPstyle headers.

5.

the assumption that all messages are equal in some way can corrupt your protocol design

Quiz 4

  4. Why are IDLs like msgpack or protobufs not necessarily a good idea?
  5. How might you relate the value of manual control for high-end professional users to some real world examples?

1.

Cheap or Nasty isn’t only about serialization strategies, it’s also about synchronous versus asynchronous, error handling and the cost of change.

2.

Most performance problems in messagebased applications can be solved by (a) improving the application itself and (b) handoptimizing the highvolume data flows.

3.

To hand-optimize your most critical data flows, you need to cheatÍ¾ to learn exploit facts about your data, something general purpose serializers cannot do.

4.

”> In most practical cases they win you little over a serialisation language backed by decent specifications written by hand or machine-generated. > Technical debt; an extra dependency. > Quite possibly, worse overall performance tha if you used Cheap or Nasty.”

5.

”> The best cards, cameras and espresso machines have manual controls. > That level of on-the-spot fine tuning often makes the difference between world class success and second best.”

Quiz 5

  2. What is the basic process you follow when writing an efficient ‘Nasty’ codec?
  3. What are some of the techniques you can use to improve your codecs?
  4. How can you use state to improve codec compression?
  5. How does knowing your data enable to improve codec performance?

1.

When you are really, truly concerned about the speed of serialization and/or the size of the result (often these contradict each other), you need handwritten binary serialization.

2.

”> Build representative data sets and test apps for stress testing. > Write a first dumb version. > Test, measure, improve and repeat.”

3.

”> Use a profiler. > Eliminate memory allocations. > Test on different platforms. > Use state to improve compression. > Know your data. > Be ready to break rules.”

4.

”> Because you’re almost certainly sending the same kinds of data many times. > There will be redundancy between instances of data. > You can detec these and use that to compress; a short value that means ““same as last time”“.”

5.

”> The best compression techniques require knowing about the data. > e.g. Methods of compressing a word list, video and a stream of stock market data area all different.”

Quiz 6

  1. How might breaking rules improve codec performance?
  2. What is GSL?
  3. What are the advantages of model-oriented code generation?
  4. What are the disadvantages of model-oriented code generation?
  5. What is a potential problem you’ll find when using ephemeral ports? What is a possible solution?

1.

”> Not using big-endian network byte order because the vast majority of platforms (x86, Android, iOS) use little-endian. > n.b. ARM uses bi-endian.”

2.

3.

”> You can create nearperfect abstractions that map to your real world. ~ This would be impossible without the freedom to tune and change the model in any way. > Perfect models quickly and cheaply. > Generate any text output from a single model. ~ Documentation. ~ Source code. ~ Test tools. > Generate perfect output because it’s cheap to improve your code generators. > Single source combining specification and semantics. > Leverage small team to massive size i.e. massive output for relatively small team size.”

4.

”> Don’t overuse. Keep it simple. Don’t get too meta. > Introduce gradually. > Put generated code into repositories.”

5.

”> ZeroMQ tries to re-establish a broken connection. > If node A stops, its peers will retry to connect. > If a new B node appears on the same port, all the peers will connect to it and continue as if its node A. > One solution would be to select a random port each time, which at least reduces the risk of hitting a just-freed port.”
