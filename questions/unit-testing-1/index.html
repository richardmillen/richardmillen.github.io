Unit Testing Quiz

! W O R K ~ I N ~ P R O G R E S S !

Quiz 1


1.

”> If you find yourself seeking esoteric mock tool features frequently, stop and take a look at your design. > Are you testing a method that’s doing too much? > Can you restructure your design in a manner so that you don’t need such complex mocking?”

2.

If your test requires extensive setup or if the production code could be holding on to stale data you might introduce a precondition assertion that verifies any assumptions your test makes in its Arrange portion.

3.

Precondition asserts suggest that the level of understanding you have about your system is insufficient. They might also suggest you’re burying too much information in setup.

4.

If you need to relax access to functions, to allow tests to verify that things actually work as intended, you shouldn’t worry most of the time.

5.

Access specifiers in tests only serve to detract from readability.

Quiz 2

  1. What is a test double?
  2. What is a test stub, when might you use one and how might it be used?
  3. What is a mock and when might one be used?
  4. What is a spy?
  5. What is a fake?

1.

”> An element that emulates a production element for testing purposes. > It replaces a dependency that exhibits one or more of the following characteristics:- ~ slow ~ unreliable ~ isn’t available (yet) “

2.

”> It’s a test double that returns a hard-coded value. > It may also ‘assert’ input from the class under test. > Inject it into the class under test, which then accesses it via an interface, or base class.”

3.

”> It is a test double that captures expectations and self-verifies that those expectations were met. > Allows you to check what objects your program is calling, with what arguments and in which order.”

4.

A test double that captures information sent to it for later verification.

5.

”> A test double that provides a light-weight implementation of a production class. > The canonical example for a fake is an in-memory (often hash-based) database. > Ensure that they are themselves unit tested. > In general, to be avoided. “

Quiz 3

  1. When might you need the more powerful features of a mocking tool?
  2. What are some key liabilities in using mocks?
  3. What should you probably do if you’re mocking the same thing in multiple places?
  4. What must you also do when using a test double and why?
  5. What is a partial mock?

1.

When writing tests around non-test-driven, poorly structured code.

2.

”> Create additional complexity. > Creates a dependency of the tests on the implementation details of the target. > Creates a dependency on the mocking tool (if one is used).”

3.

Restructure the design to eliminate this duplication.

4.

”> You must ensure that other tests cover the lines of logic that your test double supplants. > A test double represents a hole of sorts in your systems coverage.”

5.

”> When you create a mock that derives from a concrete class. > The problem is that the resulting class represents a mix of production and mocked behavior. > Usually a sign that the class you’re mocking is too large. You can likely split the class into two along these boundaries.”

Quiz 4

  1. As a general rule, how many asserts should there be per test?
  2. If you have methods that work in concert with other methods, how can you split them into multiple tests?
  3. Why might you combine multiple tests / assertions into a single test?
  4. Should a test contain conditional logic?
  5. How can you increase absraction in your tests?

1.

“One. This helps constrain size of tests.”

2.

The common logic simply forms part of the ‘arrange’ step without assertions. The assertion happens in a separate test.

3.

Having separate tests, each with a single assertion may not appear to add much value.

4.

”> No. > One behaviour per test.”

5.

”> Making them more cohesive (one assert per test). > Focus on better naming. > Abstracting away distracting details e.g. ~ Move variables into text fixture class. ~ Move logic to SetUp() and TearDown().”

Quiz 5

  1. What is a benefit of using Hamcrest-style assertions instead of simple assertions?
  2. Why do third-party libraries often cause issues when writing tests?
  3. In a multi-threaded application, how should application-specific tests be written / run?
  4. What is COVTOOL?
  5. What is the real value of a good code coverage tool?

1.

”> Improved failure messages. > Failure messages from simple assertions may not instantly tell you all you want to know.”

2.

”> Might be slow > Might require extensive configuration > Might have undesirable side effects.”

3.

”> As single-threaded. > Application code must first work in a single-threaded environment before introducing threads.”

4.

An open source test coverage tool for C++.

5.

”> Annotates your source, showing which lines are executed when your tests are run. > Knowing which lines of code aren’t covered can help you determine where you need to add tests.”
