What are the three parts of a test?


Given-When-Then / Arrange-Act-Assert

9

What do you group related tests into?


Fixtures

10

How are test fixtures arranged?


Commonly one fixture per class, but it’s not unusual to have more than one fixture for a single class, or one fixture that tests related behaviours from multiple classes.

11

Why can tests written after development can be tougher to write and maintain?


Primarily because they weren’t written with testing in mind.

12

What are the three rules of TDD?


Robert C. Martin (“Uncle Bob”) provides a concise set of rules for practicing TDD.

  1. Write production code only to pass a failing unit test.
  2. Write no more of a unit test than sufficient to fail (compilation failures are failures).
  3. Write no more production code than necessary to pass the one failing unit test.

13

What is the TDD cycle?


Red-Green-Refactor

14

What, if any issues are there with the three rules of TDD?


Rule #2 is controversial. It may not be effective if compilation requires you to wait long. You might also find it’s more effective to flesh out the design of a complete test first. It’s best employed on a case-by-case basis.

15

Adding a convenience method such as isEmpty() duplicates the functionality of size() on the side of the code’s interface. What is an effective approach to testing them both and why?


Create tests that explicitly document the link between the two concepts. e.g. IsEmptyWhenItsSizeIsZero and IsNotEmptyWhenItsSizeIsNonZero. They avoid duplication in other tests. Although they are technically unnecessary they can server to bolster the relationship between the two concepts.

16

In TDD, what rule of thumb may be used to answer the question “What’s the next test I should write?”?


Write the test that results in the simplest possible increment of production code.

17

What questions might I ask myself when coming up with new tests?


  • What list of questions can be used to decide what is the next test I should write?
  • What’s the next most logically meaningful behavior?
  • What’s the smallest piece of that meaningful behavior I can verify?
  • Can I write a test that demonstrates that the current behavior is insufficient?

18

What is the major concern when considering what should be written as a test?


  • To document public behaviours.
  • No implementation details.

19

What is the ‘London school’?


Practitioners who view TDD primarily as a design exploration tool.
Promotes the notion of Tell-Don’t-Ask.

20

What is the ‘classic school’?


Sometimes called ‘Cleveland school’.
Practitioners who emphasize verification of behaviour by inspecting state.
Avoid introducing mocks until a dependency concern forces the issue.

21

What is the FIRST mnemonic?


  • F - Fast - Short ‘specify, build, refactor’ cycle.
  • I - Isolated - From other tests. Small & focused. Fail for a single reason.
  • R - Repeatable
  • S - Self-verifying - Autonomous verification. At least one assertion.
  • T - Timely - Write them first. As incremental as possible.

Also reminds you that ‘tests come first’.

22

What is a key driver behind keeping the TDD cycle as short as possible?


The more code you grow between introducing a defect and discovering it, the more time you stand to waste in pinpointing and fixing the problem.

23

Should a happy-path test following defensive coding practices such as guard against null values?


If coded as an incremental step, then perhaps ‘YES’.
Otherwise, other tests should demonstrates the null value case, so the answer is generally ‘NO’.
Tests designed to drive happy-path behaviour should only demonstrate that behaviour.
Once a test is working it should always work, barring some future defect.

24

Should exception handling logic exist within a happy-path test?


If coded as an incremental step, then perhaps ‘YES’.
Otherwise, other tests should demonstrate the exception case, so the answer is generally ‘NO’.
Tests designed to drive happy-path behaviour should only demonstrate that behaviour.
Once a test is working it should always work, barring some future defect.

25

How could you use tests to drive improvements to a legacy system?


Write tests to characterize existing behavior.
Write tests around code you will change.
Extract code into an isolated method and test it directly.
Exposing a member may also highlight a design flaw; feature envy.

26

What steps should you take in method extraction?


  1. Add client-side call above code.
  2. Declare new method.
  3. Compile.
  4. COPY client code into method.
  5. Compile.
  6. REMOVE client code (which is now in method).
  7. Run any existing tests.
  8. Write tests for existing behaviour of method.

27

What is TPP?


Transformation Priority Premise.
Proposed by Robert C. Martin
Categorize each increment as a transformation form, each prioritized from simplest (highest priority) to most complex (lowest priority).
Choose the transformation with the highest-priority order and write the test that generates that transformation.

28

What is a key benefit of TPP?


Allows you to grow your system using the smallest possible increments.
Incrementing in transformation priority order will produce an ideal ordering of tests.

29

Why is use of the TPP an advanced topic?


Demands more up-front thought for each pass through the TDD cycle.

  • What implementation of the current test has the higher priority?
  • Does another test have higher priority than the one begging to go next?
  • What are the other tests? Maintaining a test list is almost essential.

30

What is TPL?


Transformation Priority List.

Each transform on the TPL represents a move from specific to slightly more generic.

  1. no code -> nil
  2. nil -> simple constant
  3. constant -> constant +
  4. constant -> variable
  5. statement -> statements
  6. unconditional -> conditional
  7. variable -> array
  8. array -> collection
  9. statement -> tail recursion
  10. if -> while
  11. statement -> recursion
  12. expression -> function
  13. variable -> assignment
  14. case

31

What should be willing to do in order to shorten your path to TDD mastery?


Have the will to back up and try a different route when necessary.

32

What is triangulation?


Involes approaching the same behaviour from a different angle, by adding a second test case.

33

What approach to writing a test can help clarify its intent?


Write the assertion first.
Makes you think about the goal of the behaviour you’re adding.
Grows your use of programming by intention.
Results in clearer tests.

34

What is TDD?


A software development technique used by programmers to incrementally design their system.

35

What does TDD look like?


Programmers break their work into small chunks (units).
For each unit they write a small test that shows how that code should behave.
Then they write a small chunk of code that makes the test pass.
Once the test passes, they make sure they clean up any design or coding deficiencies that they just introduced.
They use a unit test framework to help specify how every bit of added code should behave.

36

How does TDD differ from writing unit tests?


TDD uses unit tests as a way of incrementally designing the system.
Tests are written first.
Writing tests after might help the programmer find a few problems, but unit testing alone doesn’t compel programmers to change the design and quality of the codebase.

37

How does TDD change the design and quality of a system?


A few studies demonstrate that there are fewer defects when applying TDD.
It also allows programmers to continually change the code and design for the better.

38

How does TDD allow programmers to change code more than without it?


Each line of production code comes into existence only once a test exists to demontrate its behaviour.
All code under the system gets tested.
A programmer can make constant small changes as needed to keep the code clean.

39

Why does clean code matter?


Some studies show that 80% of software efforts are expended on maintaining (not fixing) the software.
As a codebase grows things that might have taken a couple of hours can take several days.

40

Why can’t programmers always write clean code in the first place?


It’s enough of a challenge to write code that simply provides expected behaviours.

  • Step 1 - Put the correct code in place.
  • Step 2 - Improve the code by making sure it’s well designed - maintainable.

Thinking of programming like writing:

  • Even good writers continually rework their sentences and paragraphs to improve understanding.
  • They still get presented with numerous problems by editors who review their work.

41

Why can’t you achieve the same results with writing unit tests after the code is completed as with TDD?


  1. For human reasons:
    • They think once the code is there, they’re done with the real work.
    • They have high confidence in their ability to write correct code, so having performed some manual tests, they have little interest in writing additional unit tests to prove what they already know.
    • They also feel the code is crafted well enough, so don’t take advantage of tests to clean up their code.
    • People in general, do as little as needed to get past ‘management mandates’.
  2. Time schedule pressures often dominate.
  3. Going back later and writing unit tests against a large existing codebase is very difficult, primarily because the codebase wasn’t structured with testing in mind.

42

Isn’t TDD wasting programmers’ time by rigid insistence on testing everything?


Most systems don’t have all that much code that’s so simple it can’t break so there’s much effort to be saved.
It’s also common for programmers to uncover defects in areas of the system that look innocuous and perfect.
Defects are very costly in many ways.
Far less time spent figuring out how the existing system behaves.

43

Why doesn’t a system only read need (say) 70% unit-tested and the rest covered with functional tests?


Are you comfortable with 30% of your system not being covered with tests that provide fast feedback?
Defects in that (nearly) third of your system will be found much later.
If you need to rework that portion to accommodate new features it will be a considerably slower effort.

44

In TDD, why don’t you end up with a lot more code, in the form of tests that you have to maintain?


Most systems are probably double the size they need to be, if not more.
Part of that is due to the inability to safely remove redundant code.
Some evidence suggests that the amount of unit testing code might be equivalent or slightly larger than a production codebase, but that codebase will likely be half the size.

45

Does TDD catch many bugs?


TDD prevents you from integrating defects into the system in the first place.

46

Why isn’t TDD a silver bullet?


TDD is a great tool, but proving that small units of code work in isolation doesn’t prove that you can string them together producing desired functionality.
You also need acceptance tests, which may also include performance tests, load tests, other forms of integration tests.
There may also be some level of exploratory testing.

47

What is the SCUMmy cycle?


aka Bad Test Death Spiral.

  1. The team writes mostly integration tests.
  2. The growing body of tests passes the pain threshold.
  3. Developers run the test suite less frequently or run subsets.
  4. Developers delete tests.
  5. Defects begin to increase.
  6. The team questions its value.
  7. The team abandons TDD.

48

What should the response be to a team / management when they question the value of TDD?


Commit to quality.

The practices of TDD as well as others such as acceptance testing, refactoring and pairing are primarly about delivering quality software in the long run, not just about minimising bugs.
It’s clear why the practices relate to quality.
Ensure that the team employs the practices in a way that helps realise the goal of quality.
Lack of concern over quality is even more unforgiving and will devastate the system.

49

What are some core reasons why most systems are a mess?


  • Because the programmers don’t have a mechanism (such as TDD) to actively ensure the code doesn’t degrade.
    Lack of education.
    • More often than not, there’s more to learn about design principles and constructs for coding well.
  • Lack of concern.
    • The system is already rife with difficult code…
  • Time pressure.
    • Just ship it…
  • Lack of review.
    • Rogue developer
    • An incoherent change
    • Inexperienced developer.
  • Lack of collaboration.
    • Diverging styles and quality.
    • Understanding another developers code can be difficult
    • Reinventing the wheel
    • Brainstorming; problem solving, solutions, ideas.

50

What is pair programming and how does it work?


Two developers actively co-develop a solution.
Typically sit side-by-side.
Each is one of two roles:

  • Driving - Coding.
  • Navigating - Providing review and strategic guidance.

Regularly swap roles during a session.
Short-lived sessions, frequent re-pairing

  • Increases knowledge, reduces risk.

51

How can pair programming help with TDD?


Developers new to TDD are more likely to revert to old, non-TDD habits without a bit of peer pressure.
Sitting with an experience TDDer can more than halve the time need to ingrain the habit of TDD.

52

What is ping-pong pairing?


The first programmer codes enough of a test to demonstrate failure then relinquishes the keyboard.
The second writes the production code and passes the test, then writes the next portion of the test (or next test).

53

What is code kata and do they work?


Test-drive the solution to a brief programming challenge.
Repeat until able to demonstrate an ideal path.
Repeat 1-3 times per week.
Seek out tweaks / refinements e.g. better API’s, language constructs.
An ideal solution == a tool in your warm-up bag.
Grow the number of katas in your bag.

54

What should be included in a teams’ TDD standards?


  • Integration standards.
    • Agree what level of testing occurs before integrating code to the SCS.
  • Test run standards.
    • How slow can tests be?
  • Disabled / commented-out tests.
    • Generally they shouldn’t be checked in without a compelling reason.
